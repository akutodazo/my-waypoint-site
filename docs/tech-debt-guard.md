# 作業パケット: 技術負債ガード（機能追加時の標準手順）

対象AI: GPT 5.6（または任意のコーディングAI）
使い方: **機能追加・修正を依頼するとき、依頼文と一緒にこの文書を渡す。**
AIは本書の手順に従って作業し、**第8節に定める6項目の形式**で成果物を報告すること。
初版: 2026-07-19（技術負債調査と解消の実績から抽出。根拠は各項の「実例」）
改訂: 2026-07-19 GPT 5.6レビュー反映（RED規律・check未定義時の扱い・弱点候補の停止位置・報告6項目化）

## 1. 前提となるプロジェクト

- **field_share**: 圃場写真共有サイト。Next.js App Router・完全静的出力・Cloudflare Workers
- **my-waypoint-site**: DJI Fly用KMZ生成PWA。Next.js・4層構造・GitHub Pages
- 共通規約: ARCHITECTURE.md（4層+依存の掟・any禁止・kebab-case・コミット接頭辞
  feat/fix/test/refactor/style/chore/docs）
- CI: lint / format:check / test / build（field_shareは + photos:check）。
  **CIを通らないコードは存在しないのと同じ扱い**

## 2. 実装前にやること

- [ ] 影響範囲を宣言する（触るファイル・触らないファイルを先に列挙）
- [ ] 既存の関数・定数・型で再利用できるものを探し、新規作成の前に必ず報告する
- [ ] 新機能はテストを先に書く（TDD: RED確認→実装→GREEN。REDとGREENは別コミット）
  - REDでは、対象テストが**期待した理由で**失敗することを確認する。RED確認前に本実装を追加しない
  - GREENでは、テストを変更せず、必要最小限の実装によって通過させる
  - RED/GREENのログまたはコミットを成果報告（第8節6項）に示す
- [ ] 既存コードへの後付けテストは「わざと壊して赤を確認→復元」で検査能力を証明する

## 3. 実装中に守ること

- [ ] **同じ数字・知識を2箇所に書かない。** ドメイン定数は集約ファイルへ:
  - field_share → `src/config.ts`（環境依存値）
  - my-waypoint-site → `src/services/flight-constants.ts`（機体・法規・地理定数）
  - 実例: 度→m換算111320が2ファイルに重複し、片方だけ直すと座標がずれる状態だった
- [ ] 層の掟: UI部品はpropsで受け取るだけ。フック・Service・Repositoryを直接importしない。
      配線はpage（画面）が行う
  - 実例: search-boxのフック直呼びを放置すると後続コードが真似て雪だるまになる
- [ ] `any` / `@ts-ignore` / `eslint-disable` は使わない
- [ ] 「壊れたら何が起きる？」の答えが怖い処理（個人情報・飛行安全・データ整合・位置精度）には
      必ずテストを付ける
  - 実例: EXIF/XMP除去（GPS流出防止）が未テストで、ライブラリ更新で黙って壊れる状態だった
  - 該当例: EXIF/XMP除去、投影計算、自動生成manifestと手動補正値(override)の分離、
    元写真と公開写真の分離

## 4. ドキュメントはコードと同じコミットで更新する

- [ ] 仕様・構成を変えたら README / CLAUDE.md の該当箇所を**同じPR内で**直す
- [ ] ドキュメントに陳腐化する固定値を書かない（例:「テスト38本」→追加のたびに嘘になる）
- [ ] 設計判断をしたら docs/decisions.md に3行（背景・決定・理由）を時系列順で追記
- [ ] 歴史的記録（進捗ログ）は書き換えない。現在形の記述だけを最新に保つ

## 5. 実装後にやること（弱点候補の提示 = 採用済みの習慣）

- [ ] 変更内容に対する**弱点候補を3〜5個、ラベル付きで提示**する。
      ラベル: `仮説` / `確認済み` / `解消済み`
- [ ] **弱点候補の提示だけで追加実装しない**（作業を止めて依頼者の採否を待つ）
- [ ] 依頼者が採用したものだけを、元PRへの追加コミットまたは別PRとしてテスト化・実装する
- [ ] **例外**: セキュリティ・個人情報・飛行安全・データ消失に直結する`確認済み`の欠陥は、
      弱点候補として後回しにせず**実装を止めて即時報告**する
- [ ] 検証の実行:
  - `npm run check` が定義されている場合は実行する
  - 未定義の場合は package.json を確認し、lint / format:check / test / build と
    プロジェクト固有の検査を個別実行する
  - checkスクリプトの新設自体は、依頼範囲に含まれる場合のみ行う

## 6. Git / PR の掟

- [ ] mainに直接コミットしない。feature/* または docs/* ブランチ + PR
- [ ] PRは1関心事1PR。コミットも関心事ごとに分割する
- [ ] マージ後はブランチを即削除する（リモート・ローカルとも）
- [ ] 既存テストの変更は仕様変更時のみ。理由をコミットメッセージに書く

## 7. やってはいけないこと

- 自動生成ファイル（photo-manifest.json等）の手編集
- field_shareの写真をphotos:importを通さず手動でpublic/photosへコピー（GPS流出リスク）
- テストが落ちたとき、原因を調べずに期待値の数字だけ合わせる
- 依頼されていない機能の先回り実装（このプロジェクト群は「ヒアリング後に追加」が方針）

## 8. 成果物の報告形式（6項目）

1. **影響範囲**: 変更したファイルと理由（1行ずつ）・変更しなかった関連ファイル・
   依頼されていないため触らなかった範囲
2. **既存資産の調査結果**: 再利用した関数・型・定数、再利用できなかった場合はその理由
3. **変更ファイルとコード本体**（diff または全文）
4. **同時更新したドキュメント**（無い場合は「更新不要の具体的理由」を書く）
5. **弱点候補リスト**（5節のラベル付き、3〜5個。対策の先回り実装はしない）
6. **検証ログ**: RED確認 / GREEN確認 / lint / format / test / build /
   プロジェクト固有検査 / 未実行項目とその理由

## 9. （付録）定期点検メニュー — 月1回 or シーズン境界に依頼する場合

機能追加時のガード（2〜7節）が主で、定期点検は網からこぼれたものを拾う保険。
以下を調査し「問題なし / 負債の芽（ファイル:行） / 負債あり」で報告すること。

- ドキュメントと実装の乖離（存在しないファイル名・古い数値・旧番号体系）
- マージ済みブランチ・未コミットファイルの放置
- 依存パッケージの更新有無と、更新が安全装置（EXIF除去等）に触れるか
- テストのデータ密結合（実データの件数・座標のハードコード）
- TODO/FIXME・any・巨大ファイル（300行超）の発生
