# 設計判断の記録（Decision Log)

判断のたびに末尾へ追記する。1エントリは「背景・決定・理由」各1〜3行。
「なぜこうなっているのか」を未来の自分とAIが3分で追えるようにするための記録。

---

## 2026-07-03 参考アーキテクチャとしてvibe-cookingを採用

- 背景: 素のHTML+JS約250行の試作サイトを本格実装する方針を決定
- 決定: vibe-cooking（Next.js+TS+層分離+TDD+CI）を手本に、5層は3〜4層に簡略化して採用
- 理由: 一人開発+学習目的では5層+DIは過剰。思想（分離・型・テスト・CI）だけを規模に合わせて移植

## 2026-07-04 KMZ生成は「ひな型複製方式」

- 背景: DJI Fly用KMZ（WPML）には未公開の設定項目が多数ある
- 決定: 1から生成せず、実機が出力した本物のKMZのPlacemarkを複製・書き換えする
- 理由: 未公開仕様（missionConfig等）を壊さず引き継げる。本物をテストの正解データにも流用できる

## 2026-07-04 Repositoryの契約は最初から非同期

- 背景: 当面の保存先はlocalStorage（同期API）だが、将来はDB移行が確定路線
- 決定: IFieldRepositoryの全メソッドをPromiseで定義
- 理由: 実装の都合ではなく将来を含めた契約として設計。移行時に呼び出し側の変更をゼロにする

## 2026-07-05 外部リソースは引数で受け取る（DI）

- 背景: 旧コードはbuildKmzの中でfetchしており、テスト不能かつ404の温床だった
- 決定: ひな型KMZはbuildKmzの引数で渡す。fetchはUI層の仕事
- 理由: テストでは本物ファイル、本番ではfetch結果と、同じ関数を両方で使えるようになる

## 2026-07-06 公開はGitHub Pages（静的書き出し）

- 背景: 公開先の選択肢はVercelとGitHub Pages
- 決定: output:'export' + GitHub Actions で GitHub Pagesへ自動デプロイ
- 理由: 既存URLの維持・運用費ゼロ・CIと同じ仕組みで学びが多い。サーバー機能が必要になったら再検討

## 2026-07-07 カメラ定数は35mm判換算で統一

- 背景: 実焦点距離と換算値が混在し、画角が実機の約半分・点数4倍で計算されていた
- 決定: DEFAULT_CAMERAを換算値（34.6×26.0/24mm）に統一。テスト期待値も仕様変更として更新
- 理由: 「換算なら換算で統一」。既存テストを変更してよい唯一の場合＝守っていた仕様自体が誤り

## 2026-07-07 KMZへの追加指定はオプトイン（後方互換）

- 背景: ジンバル角・撮影アクションをKMZに書き込む機能を追加
- 決定: buildKmzのオプション引数とし、省略時はひな型のまま変更しない
- 理由: 既存テストを変更せずに拡張するため。「省略時は何もしない」も明文化された仕様

## 2026-07-08 UIは白背景固定・視認性ファースト

- 背景: 利用場面が直射日光下のスマホ。OSのダークモードで黒背景になっていた
- 決定: ダークモードを無効化し常にライト表示。階層は色でなくサイズと太さで表現
- 理由: 屋外視認性は白背景+高コントラストが最強。デザイン性より現場の実用を優先

## 2026-07-08 テンプレート化は「機械整形→コード模範化→文書化」の順

- 背景: リポジトリを他プロジェクトの基盤（AIへ渡すテンプレート）にする方針
- 決定: Prettier導入を最初に、次にコードの例外を除去、最後に規約を文書へ切り出す
- 理由: 整形を先にすると以降の差分が本質だけになる。コードが模範形になる前に文書化すると食い違う

## 2026-07-09 配信方式は「PWAは端末で動かし、RC 2へはPC経由でKMZ差し替え」

- 背景: PWAをDJI RC 2（送信機）本体に載せて現場完結できないか検討した
- 決定: RC 2への直載せは断念。PWAはスマホ/タブレット/PCで動かしてKMZ生成し、
  USB-CでPCに繋いでRC 2の `/Android/data/dji.go.v5/files/waypoint` のダミーミッションを差し替える運用に確定
- 理由（Web検証済み）: RC 2はDJIがADBのアプリ導入権限をブロックしており**サードパーティアプリ・ブラウザを入れられない**（RC Proは可）。
  加えて `/Android/data` はAndroidの保護対象でRC 2に実用的なファイルマネージャが無いため、差し替えは実質PC必須。
  出典: mavicpilots（RC2ロックダウン/waypointフォルダアクセス）、litchiutilities import doc、dji.com RC 2 specs

## 2026-07-10 オーバーラップ0/0を全体撮影モードのスイッチにする

- 背景: 小麦圃場はオルソ用の細部撮影ではなく全体の色ムラ確認が目的。少数枚で圃場全体を写したい
- 決定: 前方・横オーバーラップがどちらも0のとき、グリッド生成ではなく高度自動計算の全体撮影に切り替える（1〜3枚で収まらない大圃場は高度149m固定で必要枚数にジグザグ分割、エラーにしない）。専用プリセット「全体撮影（色確認）」も追加
- 理由: 既存UIに新しい入力を足さず「オーバーラップ不要＝オルソ不要」という意味的に自然な条件で分岐できる。Waypointが点ごとにheightを持つ既存設計のためkmz-builder無変更で実現。DJI最小2点制約により1枚撮影は不可（2点=2枚を最小構成とする）

## 2026-07-18 ドメイン定数はflight-constants.tsに集約

- 背景: 技術負債調査で、度→m換算111320が2ファイルに重複、DJI上限200点・高度上限149m等が3ファイルに散在と判明
- 決定: 機体・法規・地理の定数（WAYPOINT_LIMIT / MAX_FLIGHT_HEIGHT_M / METERS_PER_DEG_LAT / DEFAULT_CAMERA / DEFAULT_FLIGHT_PARAMS）をservices/flight-constants.tsへ集約
- 理由: 機体変更（Lito X1導入等）・法改正時の修正点を1箇所に閉じる。片方だけ直して座標がずれる事故の予防

## 2026-07-18 UI層のフック直呼びをprops配線に統一

- 背景: search-boxがusePlaceSearchを、preset-barがFLIGHT_PRESETSを直接importし「UI部品はpropsで受け取るだけ」の掟に違反していた
- 決定: usePlaceSearchはpage側で呼びPlaceSearch型（types/domain）のpropsで渡す。プリセット一覧もpageから配線
- 理由: UIを見た目だけの部品に保つとテスト・差し替えが容易なまま。掟の違反を放置すると後続コードが真似て雪だるまになる
